# 蓝牙协议技术方案

## 1. 协议结论（基于蓝牙协议.md）
- 固定帧长：30 字节。
- 帧格式：`A5 | seq | cmd | len | data[24] | crc8 | 5A`。
- CRC8 参与字段：`seq + cmd + len + data[24]`（不含帧头尾）。
- CRC8 初始值：`0xFF`，查表方式：`crc = tab[crc ^ byte]`。
- 写配置命令：`0x11 ~ 0x1E`，应答通常 `len=1`，`data0=0x20/0x21`（成功/失败）。
- 读配置命令：同 cmd，`len=0` 作为读取请求。
- 主动上报：`0xA1`（通道值）、`0xA2`（传感器）、`0xA3`（透传/预留）。

## 2. 已发现的不一致与兼容策略
- 文档存在 OCR 污损，部分示例 cmd 与标题不一致（A1/A2/A3 段最明显）。
- 通道范围表述不统一（如“CH3-11”与“2-10”并存，疑似 0 基索引映射）。
- 兼容策略：
  1. 以“章节标题定义 + 字段语义”为主，示例帧仅作参考。
  2. 建立命令白名单与字段校验，非法值丢弃并记录日志。
  3. 对索引类字段统一做双向映射（UI: CH3~CH11 <-> 协议: 2~10）。
  4. A1/A2/A3 先按 cmd 分发，若内容结构不符再走降级解析。

## 3. 工程分层方案（适配当前 Flutter 项目）
## 3.1 传输层（BLE）
- 职责：扫描、连接、发现服务、订阅通知、写入特征。
- 建议文件：
  - `lib/src/provider/bluetooth_transport.dart`
- 关键点：
  - 写入前确认 MTU；若 `MTU-3 < 30`，做分片发送。
  - 统一输出 `Stream<Uint8List>` 原始字节流。

## 3.2 协议编解码层
- 职责：字节流拆帧、CRC 校验、命令 payload 编解码。
- 建议文件：
  - `lib/src/provider/bluetooth_frame_codec.dart`
  - `lib/src/provider/bluetooth_crc8.dart`
  - `lib/src/provider/bluetooth_command_codec.dart`
- 关键点：
  - 环形缓冲区按 `A5...5A` 取帧，长度固定 30。
  - CRC 不通过直接丢帧并计数。
  - `int8/int16` 使用补码；`uint16` 默认按文档示例用小端（如 600s -> `58 02`）。

## 3.3 会话层（请求-应答）
- 职责：帧号分配、超时重试、请求与应答匹配。
- 建议文件：
  - `lib/src/provider/bluetooth_session.dart`
- 策略：
  - 帧号自增 `0~255` 回绕。
  - 单飞行窗口（先简化），每次只发一个配置请求。
  - 超时建议 500ms，重试 2 次；仍失败则上抛业务错误。

## 3.4 业务映射层（Provider）
- 职责：协议对象与 UI 状态互转。
- 现有 provider 对接：
  - `bluetooth_provider.dart`：保留扫描/连接入口，扩展连接状态与错误状态。
  - `curve_provider.dart`、`model_selection_provider.dart`、`control_mapping_provider.dart`、`dashboard_provider.dart`：分别绑定对应 cmd。
- 建议新增：
  - `lib/src/provider/bluetooth_sync_provider.dart`（统一“读全量配置 + 增量写入”流程）。

## 4. 命令落地映射（第一批）
- 配置读写（必须）：
  - `0x11` 通道反向
  - `0x13` 中立微调
  - `0x14` 双比率
  - `0x15` 曲线
  - `0x16` 四轮转向
  - `0x17` 失控保护
  - `0x19` 模型切换
  - `0x1A/0x1B/0x1C` 混控相关
  - `0x1D` 控件分配
  - `0x1E` 系统设置
- 主动流（第二批）：
  - `0xA1` 通道实时值 -> 仪表/进度条
  - `0xA2` 传感器 -> 电压/RSSI 等
  - `0xA3` 透传 -> 先保留原始 payload

## 5. 端到端时序
1. 扫描并连接设备，发现服务与特征。
2. 打开通知流，启动帧解码器。
3. 执行“启动同步”：按页面优先级发送 `len=0` 读取命令。
4. UI 修改参数时发送写命令，收到 `0x20` 后更新本地状态。
5. 持续处理 `A1/A2` 主动上报，驱动实时 UI。

## 6. 测试方案
- 单元测试（必做）：
  - CRC8：固定输入输出回归。
  - 帧编解码：长度、头尾、CRC、补零逻辑。
  - 命令编解码：边界值（-100/0/100、0/100、通道最小最大）。
- Provider 测试（必做）：
  - 请求成功/失败状态迁移。
  - 超时重试与错误提示。
  - `A1/A2` 上报触发 UI 状态更新。
- 联调测试（建议）：
  - 抓包比对 APP 发包与协议文档字段顺序。
  - 验证掉线重连与未完成请求清理。

## 7. 迭代计划
1. M1（1-2 天）：完成传输层 + 帧编解码 + CRC8 + 单测。
2. M2（2-3 天）：接入 0x11/0x13/0x14/0x15/0x1E 与对应 provider。
3. M3（2 天）：接入混控与控件分配（0x16~0x1D）。
4. M4（1-2 天）：接入 A1/A2 流式数据 + 性能与异常收敛。

## 8. 验收标准
- 任一写配置命令成功率 > 99%（正常链路）。
- 配置页首屏同步 < 1.5s（连接后）。
- 异常帧不会导致崩溃，且有可追踪日志。
- 关键 provider 均有测试覆盖其主流程。
